# Couple Menu UX Redesign Review

结论先行：当前产品的视觉方向已经接近“情侣点菜小岛”，但交互结构还像“筛选器 + 数据维护后台”。用户第一次进入时会被大量筛选项、两个随机入口、今日推荐/决定卡片的重复关系打断，真正的核心任务“今晚我们一起决定吃什么”不够集中。

建议重构为 3 个清晰层级：
1. 今日决策：先表达两个人此刻的限制与心情，再给出一个可确认的推荐。
2. 菜单管理：维护菜品、偏好、最近吃过记录。
3. 数据工具：导入、导出、恢复示例，降级到设置/更多菜单。

本报告只给 builder 可实现的 UX / UI 规格，不要求本轮改业务代码。

## 观察范围

覆盖流程：

## 主要问题：为什么会觉得“奇怪”

### 1. 信息架构：决策和管理混在一起

现状：
- 左侧固定筛选面板同时影响“今日点菜”和“你们的小菜单”。
- 首页一打开，用户先看到大量筛选 chip，而不是“今天怎么决定”。
- 菜单管理页仍然保留完整筛选器，导致它像数据后台。

问题：
- 情侣点菜的主任务是低负担决策，不是精确检索数据库。
- 筛选是辅助，不应成为第一层主界面。

建议：
- 今日页默认只显示轻量条件选择。
- 高级筛选折叠到“更多条件”。
- 菜单页保留搜索与管理筛选，但不要让它抢今日决策的注意力。

### 2. 推荐 / 随机 / 确认关系不清

现状：
- 页面上同时有“今日推荐”卡片和“随机点菜”卡片。
- 今日推荐卡片里有“就吃这个 / 换一个”。
- 下方还有“随机点菜”。
- 点“换一个”后会出现“决定卡片”，但今日推荐卡片仍在上方，造成两个候选结果并存。

问题：
- 用户不知道“今日推荐”和“抽到的是”哪个才是最终结果。
- “就吃这个”和“标记已吃”语义接近，但一个在推荐卡，一个在决定卡，确认路径重复。

建议：
- 合并为一个主决策卡。
- 决策卡只有 3 个状态：未开始 / 候选中 / 已决定。
- 同一时间只出现一个主结果。

推荐状态流：
- 未开始：显示“准备开始点菜”，主按钮“帮我们选一个”。
- 候选中：显示菜名、理由、两人偏好、最近吃过；按钮“今晚就吃这个”“换一个”。
- 已决定：显示确认态；按钮“记为已吃”“重新选择”。

### 3. 主次按钮不稳定

现状：
- 黄色按钮同时用于“就吃这个”“随机点菜”“添加菜品”“保存”。
- 绿色按钮用于“换一个”，但它在决策流程里其实是次要动作。
- FAB 与菜单页里的“添加菜品”重复。

问题：
- 用户很难判断当前页面最重要动作。
- 移动端上 FAB 会和 toast / 底部内容抢位置。

建议按钮体系：
- Primary：只用于当前页面的核心下一步。
  - 今日页：`帮我们选一个`、`今晚就吃这个`。
  - 表单：`保存菜品`。
- Secondary：安全的替代动作。
  - `换一个`、`再选一次`、`放宽条件`。
- Ghost / Text：低频工具。
  - `导出`、`导入`、`恢复示例`、`取消`。
- Danger：破坏性动作。
  - `删除菜品`。

### 4. 筛选逻辑对普通用户不透明

现状：
- 默认偏好是“避开不想吃”。
- 菜单总数 8，但显示 7，原因不明显。
- “偏好”筛选里有“避开不想吃 / 双方都想吃 / 我想吃 / TA 想吃 / 包含不想吃”，像数据库查询条件。
- “放宽筛选”实际是重置筛选，但文案不够准确。

问题：
- 用户不知道为什么有菜被隐藏。
- “避开不想吃”和“包含不想吃”是一组开关，但被做成并列 chip。

建议：
- 今日页改成自然语言条件：
  - 谁要参与：`我和 TA 都可以` / `照顾我` / `照顾 TA`。
  - 忌口策略：默认 `避开任一方不想吃`，可切换 `今晚先不管忌口`。
  - 场景：`在家` / `外卖` / `出门` / `都行`。
  - 时间：`快一点` / `不着急` / `都行`。
- 显示候选解释：
  - `已筛出 7 道，隐藏 1 道有人不想吃的菜`。
- “放宽筛选”改成：
  - `清空条件`：重置全部。
  - `放宽一点`：只取消最严格条件，例如从“双方都想吃”退回“避开不想吃”。

### 5. 情侣双方偏好表达太机械

现状：
- 菜品卡片上“我 ♥ 想吃 / TA • 一般”是按钮。
- 点击会在 `一般 → 想吃 → 不想吃 → 一般` 三态循环。
- 没有提示这是可点击编辑，也没有二次确认。

问题：
- 三态循环是高效但不直觉，尤其“想吃”下一次变“不想吃”容易误触。
- 情侣产品的情绪表达可以更温和，但不要花哨。

建议：
- 卡片上展示为两个偏好 badge：
  - `我：想吃`、`TA：一般`。
- 点击 badge 打开小型 popover / bottom sheet，明确选择：
  - `想吃` / `一般` / `不想吃`。
- 菜单页支持批量快速标记，但默认不要用隐式循环。
- 可加一行解释：`偏好会影响推荐权重，不会公开评分。`

### 6. 添加表单过长，新增任务压力偏大

现状：
- 添加菜品表单一次展示菜名、分类、特点、获取方式、预算、准备时间、双方偏好、备注。
- 对第一次添加来说字段太多。
- “保存前可以确认一下是否已有相似菜名”只要输入菜名就出现，但没有实际相似结果。

问题：
- 用户只是想快速记录一个菜，却被迫理解数据模型。
- 提示像有查重能力，但实际没有查重反馈。

建议：
- 表单分为“必填 + 可选”：
  - 必填：菜名。
  - 推荐填写：吃法/来源、你们偏好。
  - 可选展开：标签、预算、准备时间、备注。
- 表单标题：
  - 新增：`加一道常吃的`。
  - 编辑：`编辑菜品`。
- 保存按钮文案：`加入小菜单` / `保存修改`。
- 如果没有实现查重，不要显示“相似菜名”提示；可以改成静态 placeholder：`例如：番茄牛腩饭、楼下麻辣烫`。

### 7. 菜单维护工具过早暴露

现状：
- 菜单页 toolbar 同时出现：添加、导出 JSON、导入 JSON、恢复示例。

问题：
- 导入/导出是高级工具，不应和添加菜品同级。
- “恢复示例”是破坏性操作，应更谨慎。

建议：
- 菜单页主按钮只保留 `添加菜品`。
- `导入 / 导出 / 恢复示例` 放进 `数据管理` 折叠区或右上角 `更多`。
- 恢复示例用 danger-zone 样式，并保留确认弹窗。

### 8. 移动端操作成本偏高

从 CSS 看，920px 以下会变单列，但当前结构仍是：头图 → tab → 筛选面板 → 主内容。

问题：
- 移动端用户需要先越过筛选面板，才能看到点菜结果。
- 筛选 chip 很多，会占掉首屏。
- FAB 可能遮挡底部内容，且与页面内添加按钮重复。

建议移动端结构：
- 今日页首屏顺序：
  1. 今日决策卡。
  2. 条件摘要条：`7 道候选 · 避开不想吃 · 快手优先`。
  3. `调整条件` 折叠面板。
  4. 最近吃过。
- 菜单页：
  1. 搜索框 sticky 在顶部或 toolbar 内。
  2. 菜品列表。
  3. 添加按钮使用页面内 sticky bottom CTA，不再用全局 FAB。

## Redesign UX Goal

让用户在 10 秒内完成“今晚吃什么”的决定，同时保留可维护的小菜单。

设计目标：
- 首页默认服务“决策”，不是“管理”。
- 一屏只给一个当前推荐，避免多结果并存。
- 条件表达要像日常对话，而不是数据库筛选。
- 情侣双方偏好要明确、可撤销、少误触。
- 动森/小岛风格作为轻量情绪层，不影响可读性和操作效率。

## 新页面结构

### 全局结构

建议保留两个主 tab，但改名：
- `今晚吃什么`
- `小菜单`

可选第三入口不做 tab：
- `数据管理` 放在小菜单页的折叠区或更多菜单。

Header：
- 标题：`今天吃什么？`
- 副标题：`从你们的小菜单里选一个刚刚好。`
- 状态摘要：`8 道菜 · 3 道双方想吃 · 1 道已避开`

不要在 header 里显示“7 个可选”这种依赖当前筛选但原因不明的数字；候选数应该靠近条件区。

## Screen / component structure

### A. 今晚吃什么页

推荐结构：

1. DecisionCard 主决策卡
   - 状态 A：未开始
     - 标题：`今晚想怎么吃？`
     - 文案：`我会避开任一方不想吃的菜，再从候选里挑一个。`
     - 主按钮：`帮我们选一个`
     - 次按钮：`先调整条件`
   - 状态 B：候选已生成
     - eyebrow：`推荐给你们`
     - 菜名大字。
     - 推荐理由，最多 3 条：
       - `你们都想吃`
       - `16 天没吃过`
       - `适合慢慢吃一顿`
     - meta：来源、时间、预算、标签。
     - 双方偏好：`我：想吃`、`TA：想吃`。
     - 主按钮：`今晚就吃这个`
     - 次按钮：`换一个`
     - text button：`不想看到它`（可标记本次避开，不一定写入永久偏好）
   - 状态 C：已决定
     - 标题：`今晚就吃：椰子鸡火锅`
     - 主按钮：`记为已吃`
     - 次按钮：`重新选择`
     - 反馈：`已加入最近吃过，之后几天会降低推荐权重。`

2. ConditionSummary 条件摘要
   - 显示：`候选 7 道 · 避开不想吃 · 快手优先`
   - 按钮：`调整条件`
   - 展开后显示条件面板。

3. ConditionPanel 条件面板，默认折叠
   - 场景：`都行 / 在家 / 外卖 / 出门 / 便利店`
   - 时间：`都行 / 快一点 / 普通 / 慢慢来`
   - 预算：`都行 / 低 / 中 / 高`
   - 偏好策略：
     - `避开任一方不想吃` 默认。
     - `优先双方都想吃`。
     - `照顾我想吃`。
     - `照顾 TA 想吃`。
     - `今晚可包含不想吃`。
   - 口味标签：只显示常用 6–8 个，更多折叠。
   - 操作：`应用条件`、`清空条件`。

4. RecentStrip 最近吃过
   - 展示最近 3–5 个。
   - 文案：`这些会自动降低推荐权重`。

5. EmptyDecisionState
   - 标题：`这组条件下没有菜啦`
   - 文案：`可以放宽条件，或加一道新菜。`
   - 主按钮：`放宽一点`
   - 次按钮：`添加菜品`
   - 展示当前阻塞条件：`当前限制：下馆子 + 双方都想吃 + 快手`。

### B. 小菜单页

推荐结构：

1. MenuToolbar
   - 搜索框：`搜索菜名、标签、备注`
   - 主按钮：`添加菜品`
   - 次入口：`筛选`
   - 更多：`数据管理`

2. MenuStats
   - `共 8 道菜`
   - `3 道双方想吃`
   - `1 道有人不想吃`
   - `4 道最近吃过`

3. DishList / DishGrid
   每张卡片：
   - 分类 badge。
   - 收藏星标。
   - 菜名。
   - meta：来源 / 时间 / 预算。
   - 标签，最多显示 3 个，多余 `+2`。
   - 双方偏好 badge：
     - `我：想吃`
     - `TA：一般`
   - 最近吃过：`上次 5/13` 或 `还没记录`。
   - 操作：`编辑`。

4. DataManagement 折叠区
   - `导出 JSON`
   - `导入 JSON`
   - `恢复示例数据` danger-zone。

### C. 添加 / 编辑菜品弹窗

建议不再一屏暴露所有字段。

结构：

1. Header
   - 新增：`加一道常吃的`
   - 编辑：`编辑菜品`
   - 关闭按钮。

2. 基础信息
   - 菜名，必填。
   - 获取方式：在家做 / 叫外卖 / 出门吃 / 便利店。
   - 分类：家常 / 外卖 / 下馆子 / 轻食 / 甜品 / 便利店。

3. 偏好
   - 我的偏好：想吃 / 一般 / 不想吃。
   - TA 的偏好：想吃 / 一般 / 不想吃。
   - 文案：`偏好会影响推荐权重。`

4. 可选信息，折叠
   - 标签。
   - 预算。
   - 准备时间。
   - 备注。

5. Footer
   - 新增：`取消`、`加入小菜单`。
   - 编辑：`删除菜品`、`取消`、`保存修改`。

删除：
- 保留浏览器 confirm 可接受，但更建议同风格确认弹窗：
  - `确定删除「麻辣烫」吗？`
  - `删除后不会再参与推荐。`

## Visual direction

整体保留当前温暖小岛方向，但降低装饰对信息层级的干扰。

建议保留：
- 奶油色背景。
- 木牌 header。
- 圆角 panel。
- 温和绿色和黄色。
- 手作感阴影。

建议调整：
- 不要让所有卡片都有同等厚重阴影；主卡片最高，普通列表卡片更轻。
- Emoji 只作为点缀，不作为信息传达核心。
- 不复制 animal-island-ui 的代码、图标、素材；只借鉴“圆润、木牌、岛屿、柔和边框、轻拟物按钮”的方向。
- 字体层级更稳定：
  - H1：页面品牌。
  - H2：当前任务标题。
  - 菜名：主卡片可大，列表卡不要过大。
- 色彩语义：
  - 黄色：主行动。
  - 绿色：推荐 / 成功 / 可吃。
  - 红色：不想吃 / 删除。
  - 米色：普通容器。

## Interaction states

### 推荐状态

- idle：无结果，只显示开始按钮。
- loading：按钮 disabled，文案 `翻菜单中…`，卡片可轻微摇晃；避免长动画。
- result：显示单个候选。
- confirmed：显示确认态，不再同时保留另一个推荐结果。
- empty：展示原因和恢复动作。

### 筛选状态

- 默认：`避开不想吃 + 其他都行`。
- active：条件摘要条显示所有生效条件。
- no result：明确列出导致为空的条件。
- reset：文案用 `清空条件`，不要只写 `放宽筛选`。

### 添加 / 编辑状态

- 菜名为空：保存 disabled 或使用原生 required，但要有可见提示。
- 保存成功：toast `已加入小菜单` / `已保存修改`。
- 删除成功：toast `已删除菜品`。
- 导入失败：toast 保留，最好在数据管理区显示错误说明。

### 偏好状态

- 展示：badge。
- 编辑：popover / bottom sheet 三选一。
- 成功：即时更新 badge，可短 toast：`已更新偏好`。
- 避免隐式循环，降低误触。

## Responsive behavior

### Desktop ≥ 920px

- 今日页：两栏。
  - 左/主栏：DecisionCard + RecentStrip。
  - 右/辅助栏：ConditionSummary / ConditionPanel。
- 小菜单页：toolbar 在上，grid 两列。

### Tablet 520–920px

- 单列。
- DecisionCard 必须在筛选之前。
- ConditionPanel 默认折叠。
- 菜品卡一列或两列根据宽度自适应。

### Mobile ≤ 520px

- 不建议使用全局 FAB。
- 今日页底部可用 sticky CTA：`帮我们选一个` / `今晚就吃这个`。
- 筛选 chip 改成 bottom sheet 或折叠组。
- 表单 footer sticky 在弹窗底部，方便保存。
- 数据管理默认折叠，避免误触恢复示例。

## Implementation notes for builder

### 建议的组件拆分

当前 `src/App.tsx` 已经有清晰的函数组件，可以在现有 React 结构上调整，不需要引入新框架。

建议组件：
- `AppShell`
- `HeaderSummary`
- `MainTabs`
- `TodayPage`
- `DecisionCard`
- `ConditionSummary`
- `ConditionPanel`
- `RecentStrip`
- `MenuPage`
- `MenuToolbar`
- `MenuStats`
- `DishCard`
- `PreferenceBadge`
- `PreferencePicker`
- `DishForm`
- `DataManagementPanel`
- `EmptyState`
- `Toast`

### 状态模型建议

保留现有数据模型，重点调整 UI 状态：

```ts
type DecisionState =
  | { status: 'idle' }
  | { status: 'loading' }
  | { status: 'result'; dish: Dish; reasons: string[] }
  | { status: 'confirmed'; dish: Dish; reasons: string[] }
  | { status: 'empty'; activeConditions: string[] }
```

不要同时保留 `recommendedDish` 和 `decision` 两套视觉结果。可以保留算法上的 recommendedDish，但视觉上只渲染一个 `DecisionCard`。

### 筛选模型建议

保留 `Filters`，但 UI 文案从技术字段改为用户语言。

例如：
- `preference: 'avoidHidden'` 显示为 `避开任一方不想吃`。
- `preference: 'bothWant'` 显示为 `优先双方都想吃`。
- `includeAvoid` 显示为 `今晚可包含不想吃`。

### 菜品卡建议

- 标签最多显示 3 个，避免卡片过高。
- 偏好 badge 不直接循环，点击打开 picker。
- 编辑入口保留在卡片 footer。
- 收藏可以保留星标，但 tooltip / aria-label 要明确。

### 表单建议

- 首屏只展示高频字段。
- 可选字段折叠，不隐藏已有值。
- 新增成功后可以留在当前 tab，并把新菜高亮 1–2 秒。

### 可访问性

- `dialog` 需要焦点管理：打开时聚焦菜名，关闭后回到触发按钮。
- 结果变化使用 `aria-live="polite"`。
- 不要只靠颜色表达“想吃/不想吃”，保留文字。
- chip / badge active 状态建议加 `aria-pressed`。

---

## Risks / trade-offs

1. 折叠筛选会降低高级用户的直接控制感，但能显著降低首次使用压力。
2. 合并推荐和随机结果需要轻微改状态结构，但能解决最大的信息架构问题。
3. 偏好从三态循环改 picker 会多一次点击，但能减少误触，尤其适合移动端。
4. 去掉全局 FAB 会少一个快捷入口，但能减少移动端遮挡和重复 CTA。
5. 动森/小岛风格容易变成装饰堆叠；应把风格限制在容器、边框、按钮形态，不要影响阅读。

## 建议实施顺序

P0：先修核心决策流
- 合并今日推荐、随机点菜、决定卡片。
- 今日页只显示一个主结果。
- 调整按钮文案与状态。

P1：重构筛选表达
- 今日页条件摘要 + 折叠条件面板。
- 空候选显示原因和恢复动作。

P2：优化菜单管理
- 导入/导出/恢复示例收进数据管理。
- 菜品卡偏好改为 badge + picker。

P3：移动端体验
- 决策卡优先于筛选。
- 去掉或替换全局 FAB。
- 表单 footer sticky。

P4：视觉打磨
- 降低普通卡片阴影。
- 统一按钮语义色。
- 保留轻量小岛风格，不复制外部源码/素材。
