# Couple Menu UX Redesign Review v2

结论先行：新版已经基本解决 v1 的核心问题。首页现在围绕“今晚吃什么”展开，10 秒内完成一次决策是可行的；推荐、确认、换一个、记为已吃的主流程也比旧版清楚。

本轮不建议再做大改。建议 builder 只做 4 个小修：
1. 去掉 header 里重复的“8 道菜”信息。
2. 已决定状态补一行结果说明，明确“记为已吃”会写入最近吃过。
3. 小菜单卡片里不要每张都重复“偏好会影响推荐权重”提示，改为页面级提示。
4. 移动端 tab 不要改成竖排，保持横向二分段。

---

## 观察范围

已走查线上站点：
- https://couple-menu.pages.dev

覆盖流程：
- 首屏进入。
- 点击“帮我们选一个”。
- 候选卡出现后确认“今晚就吃这个”。
- 已决定状态与“记为已吃 / 重新选择”。
- 展开“调整条件”。
- 切换到“小菜单”。
- 查看添加菜品弹窗、偏好三选一、筛选列表、数据管理入口。
- 补充查看本地代码：`src/App.tsx`、`src/App.css`。

约束确认：
- 保持动森 / 小岛风格。
- 不引入 animal-island-ui 无 license 源码或素材。
- 不改变 localStorage 数据结构兼容策略。
- 本报告只给 builder 可实现的设计结论，不改业务代码。

---

## UX goal

让情侣用户进入页面后，不先做管理、不先理解筛选器，而是在 10 秒内完成：
1. 看懂当前有多少候选。
2. 点击“帮我们选一个”。
3. 看到一个明确候选。
4. 选择“今晚就吃这个”或“换一个”。
5. 吃完后可记为已吃。

v2 已经接近这个目标。后续调整应以减噪和明确状态为主，不要再扩大功能面。

## Screen / component structure

### A. 首页 / 今晚吃什么

现状判断：通过。

当前结构为：
1. 木牌 header。
2. 双 tab：今晚吃什么 / 小菜单。
3. 主决策卡。
4. 条件摘要条。
5. 条件面板，默认收起。
6. 最近吃过。

这比 v1 的“筛选器 + 今日推荐 + 随机点菜”清楚很多。首屏核心动作只有一个“帮我们选一个”，用户无需先理解全部条件。

需要微调：
- Header 右侧现在视觉上出现两次“8 道菜”：
  - 大字：`8 道菜`
  - 小字：`8 道菜 · 3 道双方想吃 · 1 道已避开`
- 建议保留大字，下面改成：`3 道双方想吃 · 1 道已避开`。
- 或保留小字完整摘要，大字改成：`菜单概览`。

推荐 builder 采用第一种：
- `strong`: `8 道菜`
- `span`: `3 道双方想吃 · 1 道已避开`

### B. 主决策卡

现状判断：通过，小修。

状态流已经清楚：
- 未开始：`今晚想怎么吃？` + `帮我们选一个`。
- 候选中：`这次选到` + 菜名 + 理由 + `今晚就吃这个 / 换一个`。
- 已决定：`今晚就吃` + 菜名 + `记为已吃 / 重新选择`。

这条路径能在 10 秒内完成。按钮层级也正确：确认是 primary，换一个是 secondary。

需要微调：
- 已决定状态只靠标题变化和 toast 表达“确认了”。toast 会消失，卡片内缺少持久说明。
- 建议在已决定状态按钮上方或按钮下方加一行短说明：
  - `确定后可以记为已吃，之后几天会降低推荐权重。`
- 不需要新增复杂状态，也不需要弹窗。

可选文案优化：
- 候选态标题 `这次选到` 可以保留；如果想更温和，可改为 `推荐给你们`。
- 已决定态标题 `今晚就吃` 清楚，不必改。

### C. 调整条件

现状判断：通过。

新版“调整条件”已经比旧筛选自然：
- 照顾谁的口味。
- 忌口策略。
- 吃饭场景。
- 时间。

这是日常语言，不像数据库筛选。条件摘要也能解释为什么是 7 道候选。

建议不做大改。

可选小修：
- 当前条件是点选即生效。这个交互可以保留。
- 如果 builder 有余力，可在面板底部加一个 `收起条件` ghost button，移动端更好结束操作。
- 不建议加“应用条件”，否则会让用户误以为点选后还没生效。

### D. 小菜单

现状判断：基本通过，小修。

好的地方：
- 添加菜品是页面主按钮。
- 搜索、筛选、数据管理层级分开。
- 导入 / 导出 / 恢复示例已收进“数据管理”。
- 偏好从三态循环改成明确三选一，误触风险降低。

需要微调：
- 每张菜品卡都重复显示 `偏好会影响推荐权重，不是公开评分。`
- 这个提示正确，但重复 8 次会让列表显得像说明书，尤其移动端会拉长卡片。

建议：
- 在小菜单页面说明或维护面板中保留一次：
  - `偏好会影响今晚推荐，不是公开评分。`
- 菜品卡内移除这行，或只在首次/空状态显示。
- 保留三选一按钮本身，不要退回隐式循环。

### E. 添加 / 编辑

现状判断：通过。

新版表单已经把基础信息和可选信息分开：
- 基础：菜名、怎么吃、双方偏好。
- 可选：分类、特点、预算、时间、备注。

这比 v1 的长表单轻很多。新增任务压力可接受。

建议不改。

可选后续增强，不作为本轮必须：
- 移动端可把表单底部按钮做 sticky footer。
- 但当前 modal 高度和滚动已经可用，不必为了本轮重构。

### F. 数据管理

现状判断：通过。

导出、导入、恢复示例已被折叠到“数据管理”。恢复示例仍有 confirm，符合风险层级。

建议不改。

### G. 移动端

现状判断：结构方向通过，但 CSS 有一个小风险。

从 CSS 看，`@media (max-width: 620px)` 会把 `.tab-bar` 改成竖排。两个 tab 变成上下排列，会增加首屏高度，且这个产品只有两个主 tab，没有必要竖排。

建议 builder 修改：
- 移动端 `.tab-bar` 仍保持横向二分段。
- 可以减小 padding 和 gap，但不要 `flex-direction: column`。

建议 CSS 方向：
- 删除或覆盖 `@media (max-width: 620px) .tab-bar { flex-direction: column; }`。
- 保留 `border-radius: 24px` 可以。
- 两个按钮各占 50%。

## Visual direction

现状判断：通过。

当前视觉已经形成轻量小岛感：
- 木牌 header。
- 暖米色背景。
- 绿色 / 黄色主色。
- 圆角卡片、虚线边框、轻微旋转。
- 少量 emoji 作为背景情绪，不干扰操作。

建议保持克制：
- 不要继续增加动效或装饰物。
- 不要引入外部无 license 素材。
- 按钮颜色继续保持：primary 黄色、secondary 绿色、danger 红色。
- 文案继续短句，不要变成拟人化过强的游戏文案。

## Implementation notes for builder

本轮建议只做小改，不改数据结构，不改 localStorage key，不改推荐算法。

必须做：

1. Header 摘要去重
- 文件：`src/App.tsx`
- 当前：`headerSummary = ${dishes.length} 道菜 · ...`
- 建议：`headerSummary = ${bothWantCount} 道双方想吃 · ${hiddenAvoidCount} 道已避开`
- 保留 `<strong>{dishes.length} 道菜</strong>`。

2. 已决定状态增加说明
- 文件：`src/App.tsx`
- 位置：`DecisionCard` 的 `isDecided` 分支。
- 建议在按钮组前增加：
  - `记为已吃后，会出现在最近吃过里，之后几天会降低推荐权重。`
- 样式可复用普通 `p`，不要新增复杂组件。

3. 偏好提示改为页面级
- 文件：`src/App.tsx`
- 移除或隐藏 `DishCard` 内每张卡的 `preference-hint`。
- 在 `MenuPage` 的 toolbar 文案或 maintenance panel 顶部保留一次：
  - `偏好会影响今晚推荐，不是公开评分。`
- 文件：`src/App.css`
- 如果保留 class，也不要让每卡都占空间。

4. 移动端 tab 保持横向
- 文件：`src/App.css`
- 在 `@media (max-width: 620px)` 中删除：
  - `.tab-bar { flex-direction: column; }`
- 或覆盖为：
  - `.tab-bar { flex-direction: row; gap: 8px; }`

可选做：

5. 条件面板底部加“收起条件”
- 仅为移动端减少滚动成本。
- 不要加“应用条件”。当前点选即生效是对的。

不建议做：
- 不要新增第三个 tab。
- 不要把数据管理重新提到外层。
- 不要引入大型插画素材。
- 不要改变 localStorage key 或旧数据 normalize 逻辑。
- 不要把推荐结果做成列表；当前一次只给一个结果是正确方向。

## Risks / trade-offs

1. 过度修饰风险
- 当前小岛风格已经够用。
- 再增加装饰和动画会影响“10 秒内决定”。

2. 小菜单信息密度
- 明确三选一带来可理解性，但也让每张卡更高。
- 本轮用“移除重复提示”先减噪，不要改成交互更复杂的 popover。

3. 移动端首屏高度
- Header + tab + 决策卡已经占空间。
- tab 竖排会进一步挤压首屏，应避免。

4. 状态说明不足
- toast 是短暂反馈，不适合作为唯一解释。
- 已决定状态需要一行持久说明，成本低、收益高。

## Builder 必须做的改动清单

P0 / 本轮必须：
1. Header 摘要去重：不要同时出现两次“8 道菜”。
2. 已决定状态加一行“记为已吃”的持久说明。
3. 小菜单偏好提示只保留页面级一次，移除每张卡重复提示。
4. 移动端 tab 保持横向，不要在 620px 以下变竖排。

P1 / 有余力再做：
1. 条件面板底部加 `收起条件`。
2. 候选态标题从 `这次选到` 改为 `推荐给你们`，语气更柔和。

无需改动：
- 主决策流程。
- 调整条件的信息架构。
- 添加 / 编辑表单结构。
- 数据管理折叠方式。
- localStorage 兼容逻辑。
