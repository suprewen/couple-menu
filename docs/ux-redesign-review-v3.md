# Couple Menu UX Redesign Review v3

结论先行：v3 应把产品范围从“带条件筛选和数据管理的菜单系统”收回到“情侣今晚吃什么的一键决策小岛”。

保留主流程：今晚吃什么 → 帮我们选一个 → 结果 / 换一个 / 记为已吃。

删除三类噪音：
1. 首页“调整条件 / 条件筛选”相关 UI。
2. 小菜单里的任意标签 / 特点 chips。
3. JSON 导入 / 导出。

这样不会削弱核心可用性。相反，用户更少需要理解规则，页面更像一个轻量晚餐决策工具，而不是菜单数据库。

---

## 观察范围

已走查线上站点：
- https://couple-menu.pages.dev

已补充查看本地代码：
- `src/App.tsx`
- `src/App.css`
- `docs/ux-redesign-review-v2.md`

覆盖交互：

- 点击“帮我们选一个”。
- 候选态：结果、理由、换一个、今晚就吃这个。
- 已决定态：记为已吃、重新选择。
- 展开“调整条件”。
- 切换到“小菜单”。
- 查看搜索、筛选列表、数据管理。
- 打开添加菜品弹窗，并展开“补充可选信息”。

本报告只输出设计结论和 builder 可执行说明，不改业务代码。

## UX goal

让用户进入页面后，不需要先配置筛选、不需要管理标签、不需要理解 JSON 数据，就能在 10 秒内完成一次晚餐决定。

目标路径：
1. 看到“今晚吃什么”。
2. 看到当前可选数量。
3. 点击“帮我们选一个”。
4. 看到唯一结果和简短理由。
5. 选择“今晚就吃这个”或“换一个”。
6. 吃完后可“记为已吃”。

v3 的关键原则：
- 首页只服务决策，不暴露条件面板。
- 小菜单只服务维护，不提供复杂筛选。
- 菜品信息只保留能帮助识别和回忆的字段。
- 老数据兼容在读取层处理，不在 UI 上继续承载旧功能。

## 要删除的 UI / 交互清单

### A. 删除首页条件入口

删除：
- 条件摘要条：`7 道候选 · 避开不想吃 1 道 · 场景都行 · 时间都行 · 调整条件`。
- “调整条件 / 收起条件”按钮状态。
- “今晚的条件”面板。
- 面板中的四组条件：
  - 照顾谁的口味。
  - 忌口策略。
  - 吃饭场景。
  - 时间。
- “清空条件”。
- 条件改变后重置 decision 的交互。
- 空结果里由条件导致的“放宽一点”。

保留：
- 首屏候选数量。
- 推荐算法内部仍可默认避开任一方“不想吃”。
- 空结果状态，但文案改为“菜单里暂时没有可推荐的菜”。

理由：
- 用户明确要求去掉“筛选候选 / 调整条件 / 筛选候选”相关功能与 UI。
- 当前条件面板虽然比 v1 自然，但仍让产品像“先设规则再抽取”。
- v3 应让用户默认相信系统会选一个合适的，而不是让用户先调参。

### B. 删除小菜单筛选列表

删除：
- `<details><summary>筛选列表</summary>` 整块。
- 分类筛选 chips。
- 特点 / 标签筛选 chips。
- 偏好筛选 chips。
- “重置列表筛选”。
- `当前显示 8 / 8 道菜` 里的筛选语义。

保留：
- 搜索菜品。
- 添加菜品。
- 菜品列表。
- 偏好编辑。

建议替代：
- 列表摘要改为：`共 8 道菜`。
- 搜索后可显示：`搜索到 2 道菜`。
- 搜索无结果时保留空态：`没找到这道菜，可以换个关键词或添加新菜。`

理由：
- 小菜单是维护入口，不应继续承担“筛选候选”的产品心智。
- 搜索足够处理“我想找到一道菜来编辑”的轻管理需求。

### C. 删除任意标签 / 特点功能

删除：
- 数据模型面向 UI 的 `tags` 概念。
- 表单里的“特点”输入。
- 菜品卡中的任意标签 chips，例如：`热乎`、`米饭`、`下饭`。
- 推荐结果卡中的任意标签 chips。
- 搜索中对 `tags` 的显式依赖。
- `quickTags` 常量及其对应 UI。

保留但重命名：
- `category` 作为“菜系 / 类型”。
- `note` 作为“备注”。
- `sourceType` 作为“怎么吃”。
- `budgetLevel`、`timeLevel` 如 builder 认为算法仍需要，可内部保留；UI 上不再表现为标签 chips。

理由：
- 用户要求“标签”仅保留为 cuisine 和 note 两类信息，不再支持任意标签列表 / 标签 chips。
- 任意标签会把产品重新拉向数据库管理，和一键决策主线相冲突。

### D. 删除 JSON 导入 / 导出

删除：
- “数据管理” details 中的“导出 JSON”。
- “导入 JSON” label 和文件 input。
- `exportData()`。
- `importData()`。
- `onExport`、`onImport` props 链路。
- `.import-button` 相关样式。

保留：
- “恢复示例”，但不必放在“数据管理”里。
- 建议放到小菜单底部的低优先级危险区：`恢复示例菜单`。
- 恢复示例仍必须有 confirm。

理由：
- JSON 导入 / 导出对当前 MVP 是管理工具，不是晚餐决策主流程。
- 删除后不会影响 localStorage 自动保存。

## Screen / component structure

### A. 首页：今晚吃什么

最终结构：
1. Wood sign header。
2. 双 tab：今晚吃什么 / 小菜单。
3. 主决策卡。
4. 最近吃过。

不再有：
- 条件摘要条。
- 条件面板。
- 首页筛选入口。

主决策卡状态：

#### 未开始

内容：
- Eyebrow：`今晚吃什么`
- 标题：`今晚想怎么吃？`
- 说明：`我会避开你们不想吃的菜，再从小菜单里挑一个。`
- 数量：`7 道可选`
- 主按钮：`帮我们选一个`

注意：
- 这里的 `7 道可选` 来自默认推荐池。
- 推荐池默认排除任一方“不想吃”的菜。
- 不给用户暴露“调整条件”。

#### 抽取中

内容：
- 标题：`正在翻菜单…`
- 按钮：`马上就好`
- 延迟 300–500ms 可保留。

注意：
- 不增加老虎机式复杂动画。
- 动森 / 小岛感来自布局和色彩，不靠花哨动效。

#### 候选中

内容：
- Eyebrow：`候选中`
- 标题：`这次选到`
- 菜名。
- 最多 2–3 条理由。
- 元信息：怎么吃、菜系、备注短句。
- 主按钮：`今晚就吃这个`
- 次按钮：`换一个`

注意：
- 只展示一个结果。
- 不展示多个候选列表。
- 不展示标签 chips。

#### 已决定

内容：
- Eyebrow：`已决定`
- 标题：`今晚就吃`
- 菜名。
- 持久说明：`记为已吃后，会出现在最近吃过里，之后几天会降低推荐权重。`
- 主按钮：`记为已吃`
- 次按钮：`重新选择`

### B. 小菜单

最终结构：
1. 页面说明区。
2. 添加菜品按钮。
3. 搜索菜品。
4. 列表摘要。
5. 菜品卡片列表。
6. 低优先级“恢复示例菜单”。

不再有：
- 筛选列表。
- 数据管理。
- 任意标签编辑。
- JSON 导入 / 导出。

页面说明建议：
- `维护你们常吃的菜。偏好会影响今晚推荐，不是公开评分。`

列表摘要：
- 无搜索：`共 8 道菜`
- 有搜索：`搜索到 2 道菜`

### C. 添加 / 编辑菜品弹窗

最终结构：

基础信息：
- 菜名，必填。
- 怎么吃，select：在家做 / 叫外卖 / 出门吃 / 便利店。
- 我的偏好，select：想吃 / 一般 / 不想吃。
- TA 的偏好，select：想吃 / 一般 / 不想吃。

可选信息：
- 菜系，select 或短文本。
- 备注，textarea。

不再有：
- 特点。
- 任意标签输入。
- 预算 / 时间若只为 UI 展示，可删除；若算法仍用，可先内部保留默认值，不显示给用户。

建议字段顺序：
1. 菜名。
2. 怎么吃。
3. 菜系。
4. 双方偏好。
5. 备注。

理由：
- “菜系”和“备注”是用户理解菜品最轻的补充信息。
- 不再要求用户像填数据库一样维护标签、预算、时间。

## 菜品表单 / 卡片中“菜系”和“备注”的最终呈现方式

### A. 表单呈现

字段名：
- `菜系`
- `备注`

菜系：
- 推荐用 select，减少输入负担。
- 初始选项可沿用当前 `categories`：家常 / 外卖 / 下馆子 / 轻食 / 甜品 / 便利店。
- 如果 builder 想更贴近语义，可以把 `外卖 / 下馆子 / 便利店` 从菜系中移走，因为它们更像“怎么吃”；但本轮不强制重做分类体系。
- 最小改动方案：保留现有 category 值，只把 UI 文案从“分类”改为“菜系”。

备注：
- textarea。
- placeholder：`店名、做法、忌口，或者你们的小暗号。`
- 可为空。

### B. 卡片呈现

菜品卡片顶部：
- 左上角保留一个 badge：`菜系`。
- 例如：`家常`、`轻食`、`甜品`。

菜名下方：
- 展示“怎么吃”一枚轻量 pill：`在家做 / 叫外卖 / 出门吃 / 便利店`。
- 如果保留预算 / 时间，不建议继续作为多枚 tags 堆出来；最多在详情或理由里使用。

备注：
- 如果有备注，在偏好编辑下方显示一行或两行。
- 建议最多 2 行，超出省略。
- 样式比菜名弱，比 footer 强。

不再显示：
- `热乎`、`米饭`、`下饭` 等任意标签。
- 任意标签 chips 列表。

### C. 推荐结果卡呈现

结果卡中元信息建议：
- 第一行：`在家做 · 家常`
- 第二行：备注短句，仅当有备注时显示。

示例：
- 菜名：`番茄牛腩饭`
- 元信息：`在家做 · 家常`
- 备注：`适合周末多炖一点，第二天也好吃。`

不再显示：
- 预算 chip。
- 时间 chip。
- 任意 tags chip。

## Interaction states

### A. 首页无菜 / 无可推荐

状态文案：
- 标题：`小菜单还是空的`
- 说明：`先加一道你们常吃的吧。`
- 主按钮：`添加第一道菜`

- 标题：`今晚没有合适候选`
- 说明：`你们可以去小菜单调整偏好，或添加一道新的。`
- 主按钮：`去小菜单`
- 次按钮：`添加菜品`

不要出现：
- `放宽一点`
- `调整条件`

### B. 搜索无结果

文案：
- 标题：`没找到这道菜`
- 说明：`换个关键词试试，或直接添加新菜。`
- 主按钮：`添加菜品`

### C. 恢复示例

入口位置：
- 小菜单底部。
- 低优先级按钮。
- 文案：`恢复示例菜单`

确认弹窗：
- `这会用示例数据覆盖当前本地菜单，确定吗？`

## 主决策流验收标准

必须满足：
1. 首屏不出现“调整条件”“筛选候选”“筛选列表”等入口。
2. 首屏主按钮只有一个明确 CTA：`帮我们选一个`。
3. 用户不编辑任何条件，也能完成一次推荐。
4. 点击“帮我们选一个”后，只出现一个推荐结果。
5. 候选态必须有：菜名、简短理由、`今晚就吃这个`、`换一个`。
6. 点击“今晚就吃这个”后进入已决定态。
7. 已决定态必须有：`记为已吃`、`重新选择`、持久说明。
8. 点击“记为已吃”后，该菜进入“最近吃过”。
9. `最近吃过` 仍用于解释为什么避免连续几天抽到同一种。
10. 菜品结果和卡片不出现任意标签 chips。
11. 小菜单不出现 JSON 导入 / 导出。
12. 旧 localStorage 数据即使包含 `tags` 字段，页面也不崩溃；读取时可以忽略，不在 UI 呈现。
13. 旧 localStorage 数据即使来自曾经导入的 JSON，也能被 `normalizeDish` 安全处理；不合法字段使用默认值。
14. 移动端 375px 宽度下，主 CTA 在首屏可见或接近首屏可见，不被管理工具挤下去。
15. 无 console error。

## Visual direction

保持当前方向，不做大换皮。

保留：
- 木牌 header。
- 暖米色背景。
- 绿色 / 黄色主色。
- 大圆角卡片。
- 虚线边框。
- 少量食物 / 小岛氛围 emoji。

收敛：
- 删除标签 chips 后，卡片会更清爽；不需要用新装饰填补空白。
- 不新增外部插画素材。
- 不复制 animal-island-ui 无 license 源码或素材。
- 不增加拟人化过强的文案。

按钮层级：
- Primary：帮我们选一个 / 今晚就吃这个 / 记为已吃 / 添加菜品。
- Secondary：换一个 / 重新选择。
- Ghost：取消 / 关闭。
- Danger：恢复示例菜单 / 删除菜品。

## Implementation notes for builder

以下是可直接执行的改动点。

### 1. 简化类型与状态

文件：`src/App.tsx`

删除或停止在 UI 使用：
- `TodayConditions`
- `MenuFilters.tag`
- `MenuFilters.category`
- `MenuFilters.preference`
- `isConditionOpen`
- `conditions`
- `menuFilters` 中除 `search` 外的字段
- `quickTags`
- `defaultConditions`
- `defaultMenuFilters` 中非 search 字段

建议保留最小搜索状态：
- `const [search, setSearch] = useState('')`

### 2. 数据模型兼容策略

`Dish` 类型可分两步处理。

最小改动方案：
- 暂时保留 `tags: string[]` 字段，避免大范围类型改动。
- `normalizeDish()` 继续接受旧数据里的 `tags`。
- UI 不再读取 / 展示 / 编辑 `tags`。
- `saveDish()` 新建或编辑时写入 `tags: []`。

更干净方案：
- 从 `Dish` 移除 `tags`。
- `normalizeDish()` 忽略旧数据中的 `tags`。
- 但这会牵涉更多函数签名和默认数据修改。

本轮推荐最小改动方案，风险更低。

### 3. 默认推荐池

替代当前 `filterTodayDishes(dishes, conditions)`。

建议逻辑：
- 默认排除任一方 `avoid`。
- 如果排除后为空，但菜单不为空，可以返回空态，引导去小菜单调整偏好；不要提供“放宽条件”。
- 仍使用 `scoreDish()` 做权重，但去掉 `conditions` 入参。

涉及函数：
- `filterTodayDishes`
- `pickWeightedDish`
- `scoreDish`
- `buildReasons`
- `summarizeConditions` 可删除。

### 4. 删除首页条件组件

删除组件：
- `ConditionSummary`
- `ConditionPanel`
- `SegmentedGroup` 如果无其他用途。

修改组件：
- `TodayPage` 不再接收 conditions / hiddenAvoidCount / isConditionOpen 等 props。
- `DecisionCard` 空态去掉 `onLoosen`。

### 5. 简化小菜单组件

`MenuPage`：
- 删除 `filters` prop，改成 `search` 或直接传 filtered dishes。
- 删除 `onExport`、`onImport`、`onUpdateFilters`、`onResetFilters`。
- 删除“筛选列表” details。
- 删除“数据管理” details。
- 保留搜索输入。
- 将“恢复示例菜单”移到页面底部低优先级位置。

建议结构：
```tsx
<section className="panel maintenance-panel">
  <label className="field-label search-field">
    搜索菜品
    <input value={search} onChange={...} placeholder="搜菜名、菜系、备注" />
  </label>
</section>
```

列表摘要：
- 无搜索：`共 {dishes.length} 道菜`
- 有搜索：`搜索到 {filteredDishes.length} 道菜`

### 6. 简化菜品表单

`DishDraft`：
- 删除 `tagsText`。
- 把 `category` 的 UI 文案改成 `菜系`。

`DishForm`：
- 删除“特点”输入。
- 可选信息只保留：菜系、备注。
- 如果继续保留 `budgetLevel`、`timeLevel` 作为内部字段，新建时用默认值，编辑时保留旧值；不要展示给用户。

### 7. 简化卡片元信息

`DishMeta`：
- 不再 map `dish.tags`。
- 建议改为 `DishMetaLine` 或直接内联。

菜品卡片：
- 顶部 badge：`dish.category`，视觉文案解释为菜系。
- 菜名下方：`sourceLabels[dish.sourceType]`。
- 备注：保留 `dish.note`。

结果卡：
- 元信息：`sourceLabels[dish.sourceType] · dish.category`。
- 备注：有则展示。

### 8. 搜索逻辑

修改 `filterMenuDishes`：
- 只按 `dish.name`、`dish.category`、`dish.note`、`sourceLabels[dish.sourceType]` 搜索。
- 不再搜索 `tags`。
- 删除 category / tag / preference 的筛选条件。

### 9. 删除 JSON 导入导出

删除：
- `exportData()`。
- `importData()`。
- 对应 props。
- `Blob` 下载逻辑。
- 文件 input。
- `.import-button` CSS。

保留：
- `loadDishes()`。
- `parseDishArray()`。
- `normalizeDish()`。

注意：
- 删除导入 / 导出不等于删除本地保存。
- localStorage 自动保存仍是核心数据能力。

### 10. CSS 清理

文件：`src/App.css`

删除或清理：
- `.condition-summary`
- `.condition-panel`
- `.segmented-group`
- `.chip-group`
- `.import-button`
- `.import-button input`
- 只服务 tags chips 的 `.tag-list` 样式可重命名或收敛为 `.meta-line`

保留：
- `pref-editor`
- `preference-choice`
- `dish-note`
- modal / form 基础样式。

移动端：
- 当前 CSS 已没有 v2 提到的 tab 竖排问题；继续保持 tab 横向即可。

## Risks / trade-offs

### 1. 删除条件后，用户少了一些控制感

取舍：接受。

这是本轮范围收敛的核心。产品应默认替用户处理日常权重，而不是要求用户每晚调条件。

缓解：
- 保留小菜单中的双方偏好。
- 保留“换一个”。
- 保留“最近吃过”降低重复推荐。

### 2. 删除标签后，推荐理由可能变少

取舍：接受。

理由可以来自：
- 双方偏好。
- 最近吃过时间。
- 收藏。
- 怎么吃。
- 备注。

不需要靠任意标签堆信息。

### 3. 删除 JSON 导入 / 导出后，迁移能力下降

取舍：接受。

当前 MVP 更重要的是低认知负担。localStorage 自动保存仍存在。未来如果真的需要迁移，可做成隐藏高级入口或独立设置页，而不是小菜单主界面功能。

### 4. 旧数据兼容要谨慎

必须做：
- 旧数据有 `tags` 不崩溃。
- 旧数据缺少 `category` / `note` 不崩溃。
- 旧数据字段类型不合法时使用默认值。

建议 builder 保留 `normalizeDish()` 的防御性写法，并补一组旧数据测试。

---

## Final recommendation

v3 不需要重新设计视觉。

要做的是信息架构减法：
- 首页只保留一键决策。
- 小菜单只保留搜索、添加、编辑、偏好、恢复示例。
- 菜品信息只保留菜名、怎么吃、菜系、双方偏好、备注、最近吃过。
- 任意标签、筛选条件、JSON 导入导出全部退出 UI。

这会让 Couple Menu 更像一个每天能用的小岛工具，而不是一个需要维护规则的菜单后台。
